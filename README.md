###### Подготовка сервиса к запуску

Создать `.env` из примера и заполнить переменные:
```
cp .env.example .env
```

###### Порядок первого запуска

```
make init
```

Команда последовательно выполнит: `keys` → `build` → `up`

Миграции применяются автоматически при запуске через отдельный контейнер до старта приложения.

После того как сервисы поднялись, создать суперпользователя:
```
make superuser
```

###### Команды

Генерировать RSA-ключи для подписи JWT (пропускает, если файлы уже существуют):
```
make keys
```

Собрать Docker-образы:
```
make build
```

Запустить контейнеры в фоне (удаляет осиротевшие контейнеры):
```
make up
```

Пересобрать образы и перезапустить контейнеры без удаления томов:
```
make rebuild
```

Применить все pending-миграции:
```
make migrate
```

Создать суперпользователя (интерактивный ввод email и пароля):
```
make superuser
```

Остановить контейнеры без удаления томов:
```
make down
```

Остановить контейнеры и удалить все тома (данные БД будут потеряны):
```
make down-v
```

Логи контейнера приложения:
```
make logs
```

Применить миграции и сгенерировать новую по текущему состоянию моделей:
```
make revision name=описание
```

Сгенерировать новую миграцию без предварительного `upgrade head` (для чистой БД):
```
make revision-fresh name=описание
```

Полный сброс и повторный запуск — удаляет тома, пересобирает, поднимает, мигрирует, создаёт суперпользователя:
```
make fresh
```

###### Работа с ролями и правами

Модель доступа: пользователь → роли → права. Список прав конкретного пользователя возвращает отдельный эндпоинт. Права не включаются в JWT — токен содержит только `sub` (user_id) и `is_superuser`.

Все операции управления ролями и правами доступны только суперпользователю. Перед выполнением запросов необходимо получить access-токен через `POST /api/v1/login/` и передавать его в заголовке:

```
Authorization: Bearer <access_token>
```

Access-токен живёт ограниченное время (задаётся в `ACCESS_TOKEN_EXPIRE_MINUTES`). Когда он истечёт, получить новый можно через `POST /api/v1/refresh/`.

**Роли**

Создать роль:
```
POST /api/v1/roles/
{
  "name": "subscriber",
  "description": "Пользователь с подпиской",
  "is_active": true
}
```

Список всех ролей:
```
GET /api/v1/roles/
```

Роль с привязанными правами:
```
GET /api/v1/roles/{role_id}/
```

Изменить роль (все поля опциональны):
```
PATCH /api/v1/roles/{role_id}/
{
  "name": "new_name",
  "description": "Новое описание",
  "is_active": false
}
```

Удалить роль:
```
DELETE /api/v1/roles/{role_id}/
```

Системные роли (`is_system=True`) удалить нельзя — вернётся `409 Conflict`.

Назначить роль пользователю:
```
POST /api/v1/roles/{role_id}/users/{user_id}/
```

Снять роль с пользователя:
```
DELETE /api/v1/roles/{role_id}/users/{user_id}/
```

Добавить право к роли:
```
POST /api/v1/roles/{role_id}/permissions/{permission_id}/
```

Убрать право из роли:
```
DELETE /api/v1/roles/{role_id}/permissions/{permission_id}/
```

**Права**

Создать право:
```
POST /api/v1/permissions/
{
  "code": "movie:watch_premium",
  "name": "Смотреть премиум-контент",
  "description": "Доступ к фильмам по подписке Premium",
  "category": "movies"
}
```

`code` — уникальный ключ в формате `область_системы:действие`, максимум 100 символов. Другие сервисы должны запрашивать список прав пользователя через `GET /api/v1/users/me/permissions/`,получать набор кодов и проверять наличие нужного кода в нём. Примеры: `movie:watch`, `movie:watch_premium`, `content:upload`, `billing:refund`, `admin:manage_roles`.

`category` — группа для группировки при отображении списка прав, максимум 50 символов. По умолчанию `general`. Примеры: `movies`, `billing`, `content`, `system`.

Список всех прав:
```
GET /api/v1/permissions/
```

Изменить право:
```
PATCH /api/v1/permissions/{permission_id}/
{
  "code": "movie:watch_premium",
  "name": "Новое название",
  "description": "Новое описание",
  "category": "movies"
}
```

Удалить право:
```
DELETE /api/v1/permissions/{permission_id}/
```

**Проверка прав пользователя**

Возвращает список прав текущего пользователя. Доступно любому авторизованному пользователю:
```
GET /api/v1/users/me/permissions/
```

- **Обычный пользователь** — возвращаются только права, привязанные через его роли.
- **Суперпользователь** — возвращаются все права из БД, независимо от назначенных ролей.

Суперпользователю не нужно вручную назначать роли — он автоматически получает полный набор прав. Другие сервисы при этом работают по единой логике: вызывают этот эндпоинт и проверяют наличие нужного `code` в ответе, не обрабатывая флаг `is_superuser` отдельно.
